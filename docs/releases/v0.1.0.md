# Release notes — v0.1.0

Release date: **2026-04-03**

`v0.1.0` is the first polished public GitHub release of `rl-developer-memory`.

## Highlights

- public-facing repository structure cleaned up for GitHub publication
- Python/module naming normalized with compatibility shims to avoid abrupt import breakage
- large maintenance and release-readiness surfaces split into more readable modules
- tests reorganized into clearer unit/integration/regression/smoke domains
- installation, usage, dependency, and validation docs simplified and aligned
- GitHub community, security, CI, and release surfaces hardened

## Repository and codebase improvements

- `EXAMPLES/` renamed to `examples/`
- `docs/theory_to_code.md` renamed to `docs/THEORY_TO_CODE.md`
- clearer canonical module/script names:
  - `mcp_reuse_harness`
  - `release_readiness`
  - `quality_checks`
  - `skill_bundle_sync`
- compatibility shims kept for older internal import paths
- `maintenance.py` split into a façade plus focused CLI modules
- release-readiness logic split into orchestration and lower-level checks

## Documentation and GitHub polish

- README rewritten for clearer public onboarding
- support, security, contribution, and code-of-conduct surfaces completed
- issue templates and PR template aligned to public usage
- release workflow hardened with tag/version verification and clean-wheel install checks
- CI matrix expanded to Python `3.10`, `3.11`, and `3.12`

## Compatibility notes

- No intentional public API break is introduced in this release.
- Some internal modules/scripts were renamed, but compatibility shims were kept to reduce transition risk.
- The release-readiness report may still show **active rollout = no-go** without live shadow-soak evidence; this is expected operational policy, not a packaging/repository release failure.

## Known non-blocking notes

- `python -m rl_developer_memory.maintenance --help` may emit a non-blocking `runpy` warning in some validation contexts.
- Example result files under `examples/results/` are committed reference snapshots; local example runs should prefer temporary output paths.

## Recommended next steps after publishing

- enable GitHub private vulnerability reporting in repository settings
- confirm `main` remains the only long-lived branch
- protect `main` with required CI checks if desired
- use short-lived PR branches for future work and delete them after merge
